# design-sync notes — Psihointegritet Design System

## Why this sync looks unusual

`frontend/` is a Next.js **app**, not a publishable component-library package —
there's no `dist/` + `.d.ts` build to point the converter at. The whole setup
below exists to make the `package` (non-storybook) shape's "no dist, synthesize
from src/" fallback work against an app instead of a library.

## The scoped scratch package

`.design-sync/prep-scoped-pkg.mjs` regenerates
`frontend/node_modules/psihointegritet-ds/` before every build (wired as
`cfg.buildCmd` — re-run it before `package-build.mjs` on every re-sync). It:

- Copies `ui/`, `shared/`, `panel/`, `motion/` from `frontend/src/components/`
  verbatim (real copies, not symlinks — the converter's `walk()` skips
  symlinked directories since `readdirSync` Dirent doesn't report a symlink
  as `isDirectory()`, so a curated symlink tree silently contributes zero
  files).
- Rewrites `@/components/(ui|shared|panel|motion)/X` cross-references to
  relative paths into the copies (`../ui/X`) instead of the tsconfig alias —
  otherwise they'd resolve to the REAL frontend/src files, which still import
  the unpatched `next/image`/`next/link`/`@clerk/nextjs` and reintroduce every
  fix below.
- Writes `tsconfig.alias.json` (a minimal `{baseUrl, paths}` file, not the
  real `frontend/tsconfig.json`) for `cfg.tsconfig`. **Do not point cfg.tsconfig
  at the real tsconfig.json** — `lib/bundle.mjs`'s comment-stripping regex
  (`/\*[\s\S]*?\*\//`) misfires on the `"**/*.ts"` globs in tsconfig's
  `include` array: the `/*` in `"@/*"` pairs with the `*/` inside `**/*.ts`
  and the regex eats everything between as a fake block comment, corrupting
  the JSON before `JSON.parse`. This is a bug in the shared lib script (not
  forkable per skill rules) — the workaround is a glob-free tsconfig instead.
- Compiles `frontend/src/app/globals.css` (Tailwind v4, `@import "tailwindcss"`)
  with the real `@tailwindcss/postcss` engine so `cssEntry` ships actual
  utility classes, not just the `@theme` token declarations (Tailwind v4 has
  no static precompiled stylesheet otherwise).
- Prepends a `:root{--font-instrument-sans:"Instrument Sans";--font-newsreader:"Newsreader";}`
  block — `next/font/google` sets these vars via a generated `.variable`
  class on `<html>` at runtime; there's no static rule for them anywhere in
  source.
- Shims `next/image` → plain `<img>`, `next/link` → plain `<a>` (both pull in
  server-only Next internals — `fs`/`stream`/`zlib`, bloom-filter/gzip-size —
  when bundled with `platform:'browser'`), and `@clerk/nextjs` → a fake
  `useClerk`/`useUser` pair (its root barrel re-exports server + app-router
  internals alongside the client hooks; importing anything from it drags in
  the same server-only graph). `LogoutAvatarMenu` renders as a signed-in user
  with no photo (initials shown, not a real avatar) as a result.
- `componentSrcMap: {"Image": null, "Link": null}` in config excludes the
  shim default-exports themselves from being discovered as components (they
  live under `src/_shims/` which is inside the walked `srcDir`).

## Brand fonts (extraFonts)

`.design-sync/fonts-src/{instrument-sans,newsreader}.css` + their `.woff2`
files were harvested from a prior `next dev` build's
`frontend/.next/dev/static/{chunks,media}/` output (real Google Fonts,
self-hosted by `next/font`, OFL-licensed). **Re-sync risk**: if `frontend/.next`
is ever cleaned and no dev/build server has run since, these files aren't
regenerated by `prep-scoped-pkg.mjs` — it only reads from
`.design-sync/fonts-src/`, which IS committed, so this only matters if that
directory needs to be regenerated (e.g. a font upgrade). To regenerate: run
`npm run dev` in `frontend/` once, then re-harvest from
`frontend/.next/dev/static/chunks/[next]_internal_font_google_*.css` and the
referenced `../media/*.woff2` files, stripping the `../media/` prefix from
the `url()`s since the woff2s are copied next to the CSS.

## Scope

Only `ui/`, `shared/`, `panel/`, `motion/` are synced (~27 components) —
`sections/` (hero, site-header, faq, therapist directory, etc.) are
page-composition blocks, not independent design-system pieces, and were
excluded by Milan's explicit choice at kickoff (2026-07-24).

## Declaration emit (`cfg.tsconfig` alone isn't enough for real props)

`prep-scoped-pkg.mjs` also runs `tsc --emitDeclarationOnly` (via
`tsconfig.emit.json`, written into the scratch package) into
`psihointegritet-ds/types/`. Without this, ts-morph's prop extractor
(`propsBodyFor`) has zero `.d.ts` files to search — the "no dist" package
shape ONLY looks at `**/*.d.ts` under the types root, never at `.tsx` source
directly — so every component's props stub to `[key: string]: unknown`. 25/27
components already name a `<Name>Props` interface by convention, so real
declarations (with actual prop unions, e.g. `StatusBadgeProps.tone`) come out
cleanly; the tsc pass reports ~19 non-blocking type errors (path-alias edges
it can't fully resolve) but still emits.

**`rootDir` must cover both trees, not just our copied `src/`.** An earlier
version set `rootDir: 'src'` (psihointegritet-ds's own copy only) — since the
`@/*`-aliased cross-references (`@/helpers/cn`, `@/content/homepage`, etc.)
pull the REAL `frontend/src/**` files into the same compile graph, tsc threw
`TS6059: File ... is not under 'rootDir'` for every one of them, and because
`noEmitOnError` defaults to false, it still emitted their declarations
anyway — landing 17 stray `.d.ts` files beside the real source (under
`content/`, `helpers/`, `lib/content-governance/`, `types/`) since there was
no valid rootDir-relative outDir path to compute. Fixed by setting
`rootDir: '../..'` (the common ancestor of `psihointegritet-ds/src` and the
real `frontend/src` is `frontend/` itself, two levels up from `PKG_DIR`) —
cross-referenced files now land safely at `types/src/...` and our own copied
components nest one level deeper at
`types/node_modules/psihointegritet-ds/src/...` (ugly, but
`package-build.mjs`'s prop extractor walks the whole `types/` tree
recursively and matches by interface NAME, so the extra nesting is inert).
If you ever see untracked `.d.ts` files under `frontend/src/` after running
`prep-scoped-pkg.mjs`, this regression is back — check `rootDir` first.

## Preview authoring — completed 2026-07-24

26/27 components have authored previews (`.design-sync/previews/*.tsx`), all
graded `good`, all real content ported from actual usage (never invented) —
FAQ items, therapist bios, section copy, panel status strings, all sourced
from `frontend/src/content/*.ts` and real call sites under
`frontend/src/components/sections/**` and `frontend/src/app/**`.

**BackToSiteMenuItem is the one floor-card exception** — permanent, not a gap
to close later. Its only content is a HeadlessUI `<MenuItem>`, which renders
nothing without a `<Menu>` ancestor providing React context. The preview and
the exported component are bundled by SEPARATE esbuild passes, so each
carries its OWN independent `@headlessui/react` module instance with its own
`Context` object identity — a `<Menu>` composed in the preview can never
satisfy the `<MenuItem>` context read inside the bundle's export. Confirmed
via direct Playwright console capture: `"<Menu.Item /> is missing a parent
<Menu /> component."` This is structural (no per-package "share this
dependency across bundles" option exists in the converter) — don't attempt
to fix it in a future sync without a different bundling strategy.

**Two components needed `cfg.overrides` for card framing** (in
`.design-sync/config.json`): `StickyBar` and `ConfirmModal` both use
`position: fixed` — package-validate flagged `[GRID_OVERFLOW]`
("stories position content outside their cells (fixed/portal)"), fixed with
`{"cardMode": "single", "primaryStory": "...", "viewport": "..."}`.
`SectionHeading`, `Reveal`, `TherapistRow` got `{"cardMode": "column"}` for
being wider than a grid cell.

**Two animation-timing traps hit during authoring** (both fixed at the
preview-authoring layer, not config):
- `Reveal` uses `motion/react`'s `whileInView` (IntersectionObserver) — a
  static capture screenshot doesn't wait for the observer to fire or the
  0.7s transition to finish, so the first attempt captured fully blank
  (opacity 0). Fix: `Reveal` itself renders a plain unanimated `<div>` when
  `useReducedMotion()` is true — the preview overrides `window.matchMedia`
  for the `prefers-reduced-motion` query to force that branch, sidestepping
  the animation instead of racing it.
- `StickyBar` needed BOTH a `transition-duration:0` override (same
  screenshot-vs-animation race) AND a fake scroll signal: the capture card's
  single-story wrapper is `.ds-single{transform:translateZ(0)}`, and a
  `transform` on an ancestor makes it the CSS containing block for
  `position:fixed` descendants — so real page-scrolling and "rendered in
  frame" become mutually exclusive for any `position:fixed` component
  through this harness (scrolling far enough to cross the component's own
  `scrollY > 50` check also pushes it out of the screenshot). Fix: override
  `window.scrollY` via `Object.defineProperty` and dispatch a synthetic
  `scroll` event so the component's listener flips its internal state
  without the page (or the pill's rendered position) ever actually moving.
  **If you ever see a `position:fixed` component's preview blank or
  clipped, check this first** — it'll recur for any future fixed/portal
  component synced from this app.

Two RENDER_THIN warns are expected and benign, not bugs:
`ConfirmModal` and `JsonLd` both measure 0px rendered height even though
content is visibly present (the same fixed/portal-vs-transform quirk above;
`JsonLd` additionally paints nothing by design — it's a `<script
type="application/ld+json">`, structured data with no visual output at all).

## Re-sync risks

- `frontend/node_modules/psihointegritet-ds/` is entirely regenerated
  machine state (gitignored) — never hand-edit it, edit the real
  `frontend/src/components/*` files or `prep-scoped-pkg.mjs` instead.
- The Clerk/next shims mean `LogoutAvatarMenu`'s preview can never show a
  real avatar photo or exercise real sign-out — that's a deliberate,
  permanent limitation of syncing an app (not a library) this way, not
  something a future sync should try to "fix".
- If `frontend/tsconfig.json`'s `paths` change (new alias, different target),
  update `tsconfig.alias.json`'s generation in `prep-scoped-pkg.mjs` to match
  — it's hand-mirrored, not derived.
- `.design-sync/conventions.md` (wired via `cfg.readmeHeader`) documents the
  real Tailwind token/class vocabulary actually shipped by this sync's scope
  (colors, radii, shadows, fonts) — every name in it was verified against the
  compiled `_ds_bundle.css`. If the scoped folders change, re-verify before
  trusting it (a token used only in `sections/` won't be in this build's CSS
  even if it exists in the wider app).
